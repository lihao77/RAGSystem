# 应急决策能力扩展路线图

## Summary

当前系统已经具备进入“能力建设阶段”的条件，不需要再做一轮大规模基础架构重构。

结论是：

- 主线基础已经够用，可以开始工具开发、Skill 开发和 MCP 集成。
- 但不建议完全跳过基础功能补强，还需要补一层“应急决策承载面”，否则能力会停留在问答和零散工具调用，难以形成稳定决策助手。
- 目标应按“少量底座补强 + 第一批决策能力并行开发”推进，而不是先做纯架构，也不是立刻做重型联动指挥平台。

默认目标按以下定义收敛：

- 形态：决策助手
- 数据：静态知识与实时业务数据并重
- 范围：生成研判、预案建议、处置建议、依据链路，不直接做高风险自动执行

## Key Changes

### 1. 先补最小必要底座，不再继续大重构

补的不是通用架构大改，而是应急决策专用的最小承载能力：

- 定义统一的“决策对象”输出，而不只返回自然语言答案。
- 定义统一的“态势/证据/建议/行动项”结构，作为 Agent、Tool、Skill、MCP 的共同中间层。
- 给前端补一个轻量工作台容器，至少能同时承载：
  - 决策结论
  - 证据依据
  - 可视化态势
  - 待办行动项
- 固化决策链路的可观测字段，至少记录：
  - 使用了哪些预案、规则、实时数据
  - 哪些建议来自规则，哪些来自模型归纳
  - 哪些字段缺失导致置信度下降

建议新增一套统一对象，名称可直接采用：

- `SituationSummary`
- `EvidenceRecord`
- `DecisionRecommendation`
- `ActionTask`
- `DecisionPackage`

最小要求：

- 所有应急能力最终都能落到 `DecisionPackage`
- 自然语言回答只是 `DecisionPackage` 的一种呈现，不再是唯一产物

### 2. 优先进入工具开发，而不是先扩 Agent 数量

下一阶段主力应放在工具层，因为当前瓶颈不是“不会编排”，而是“没有足够强的决策输入与规则执行能力”。

第一批应急工具建议直接开发为静态 Tool 或 MCP Tool：

- `get_incident_snapshot`
  - 聚合雨情、水情、灾情、风险点、行政区、时间窗口，输出标准态势对象
- `match_emergency_plan`
  - 按事件类型、等级、区域、阶段匹配适用预案和条款
- `assess_response_level`
  - 根据规则判断建议响应等级、升级条件和触发依据
- `generate_action_checklist`
  - 按场景输出行动建议清单、责任角色、先后顺序、前置条件
- `query_resource_status`
  - 查询物资、队伍、避险点、医院、设备等状态
- `estimate_impact_scope`
  - 根据行政区、人口、设施、历史灾情或实时数据估计影响范围
- `build_decision_brief`
  - 把结构化对象整理成适合指挥和值班场景的简报

原则：

- 规则明确、数据依赖强、需要稳定复用的能力优先做 Tool
- 外部系统已有接口、且不适合内置维护的能力优先做 MCP Tool
- 临时性、流程性、提示型知识优先做 Skill

### 3. Skill 负责“方法论与处置模板”，不是替代工具

Skill 在应急决策里应承担这些角色：

- 场景化 SOP
  - 如暴雨内涝、江河洪水、水库险情、山洪、城市积涝、转移安置
- 报告模板
  - 值班快报、会商摘要、处置建议、上报材料、指令草案
- 决策框架
  - “先判态势，再判等级，再判处置，再判资源缺口，再判升级条件”
- 专家知识压缩
  - 例如预警分级解释、典型联动流程、术语口径统一

建议先做 4 类 Skill：

- `flood-situation-assessment`
- `emergency-plan-interpretation`
- `response-checklist-generator`
- `decision-brief-writer`

Skill 不应直接承担强数据融合或强规则判定；这部分应下沉到 Tool 或 MCP。

### 4. MCP 集成应成为实时数据主入口

由于目标是“静态知识 + 实时业务系统都重”，MCP 应尽快接入，而不是后置。

第一批 MCP Server 建议按系统边界接入：

- 气象、雨情 MCP
- 水文、水库、河道 MCP
- 灾情上报 MCP
- 物资与队伍 MCP
- 避险点、人口转移 MCP
- 地图与空间分析 MCP
- 文档与制度库检索 MCP

接入顺序建议：

1. 只读查询型 MCP
2. 聚合统计型 MCP
3. 空间查询型 MCP
4. 最后才考虑写操作或联动型 MCP

要求：

- 所有 MCP 输出先归一化为统一结构，再进入 Agent 推理
- 不让 Agent 直接消费五花八门的外部原始 payload
- 高风险写操作默认不纳入当前阶段

### 5. Agent 结构保持轻量，不建议现在再拆很多 Agent

当前建议保留少量角色清晰的 Agent：

- `orchestrator_agent`
  - 负责任务编排和结果汇总
- `emergency_agent`
  - 升级为“应急决策助手”，不再只是预案问答
- `chart_agent`
  - 负责态势图、地图、可视化
- 可选新增一个 `situation_agent`
  - 若后续发现“态势研判”与“预案建议”明显分离，再拆

当前不建议一开始就拆出大量专业 Agent，如“雨情 agent、水情 agent、物资 agent、安置 agent”。

先用工具和 Skill 把能力做实，后续再根据复杂度拆 Agent。

## Implementation Changes

### 数据与接口

应新增一个统一的决策输出协议，至少包含：

- `situation`
  - 当前态势摘要、时间范围、区域、事件类型、关键指标
- `evidence`
  - 依据列表，标记来源类型：预案、规则、实时系统、人工输入
- `recommendations`
  - 建议事项，带优先级、适用条件、理由
- `actions`
  - 可执行动作，带责任角色、时序、依赖、风险提示
- `visualizations`
  - 可关联现有 artifact 或 presentation
- `confidence`
  - 整体和分项可信度
- `gaps`
  - 当前缺失的信息和待确认项

对外行为变化：

- `emergency_agent` 默认输出不再只是一段文本，应优先产出结构化决策包，再渲染成文本。
- 可视化结果应作为决策包的一部分挂载，而不是独立散落输出。
- 后续任何应急 Tool 或 MCP 接入，都必须声明它产出的是哪个结构对象片段。

### 前端与工作台

前端不需要先做完整指挥平台，但至少要补一个轻量决策工作台区域：

- 左侧聊天保留
- 右侧展示 `DecisionPackage`
- 底部可选展示证据、事件、可视化、诊断

最小可见能力：

- 看到结论
- 展开证据来源
- 看到推荐动作清单
- 看到地图或图表
- 看到“还缺什么信息”

### 观测与评估

必须同步补：

- 决策链路日志
- 引用依据回显
- 工具和 MCP 调用轨迹
- 错误分类
  - 无数据
  - 数据冲突
  - 无适用预案
  - 规则未命中
  - 输出不完整

评估集建议围绕场景构建，而不是围绕单问答构建：

- 暴雨红色预警 + 城区积涝
- 河道超警 + 水库泄洪
- 山洪危险区群众转移
- 多地灾情上报不一致
- 资源不足时的处置建议

## Test Plan

必须覆盖以下测试：

- Tool 单测
  - 态势聚合、预案匹配、响应等级判断、行动清单生成
- MCP 合同测试
  - 输入参数、输出归一化结构、异常返回
- Skill 测试
  - SOP 场景是否能产出稳定模板和结构
- Agent 集成测试
  - 给定一个场景，能否输出完整 `DecisionPackage`
- 前端验收
  - 能否在一次会话中同时展示决策文本、证据、行动项、可视化
- 回归测试
  - 普通问答、预案查询、可视化链路不退化

关键验收标准：

- 系统能对一个具体灾情场景输出结构化决策建议，而不是只给泛泛文字
- 决策建议能明确列出依据和信息缺口
- 至少一类实时数据源能稳定接入并参与决策
- 至少一类地图或图表能挂接到决策输出
- 无外部实时数据时，系统仍可退化为“预案 + 历史知识”的辅助决策模式

## Assumptions

- 当前主线架构已足够支撑能力扩展，不再进行大规模底层重构。
- 当前阶段不做高风险自动处置，不默认触发外部写操作。
- MCP 主要承担实时业务数据接入，Skill 主要承担方法论和模板，Tool 主要承担规则与结构化处理。
- 若后续目标升级为“联动指挥平台”，再追加更重的工作台、事件流、状态机和审批闭环设计。
